# Capability Registry — Reference Specification

> **Single source of truth** for what tools, agents, skills, MCP servers, and dependencies are available to SPEAR phases.

Every SPEAR phase consults the capability registry before designing solutions. The registry prevents building what already exists and ensures the right tool is used for each task.

---

## Registry File

Location: `.spear/capability-registry.json`

Generated by the `/spec` command (Step 1) and refreshed by `/plan` (Step 1). Can be manually refreshed with `/spear:toolkit refresh`.

## Discovery Sources

The registry is built by scanning these sources (in order):

| # | Source | How to Scan | What It Produces |
|---|--------|-------------|-----------------|
| 1 | **Claude Code Skills** | Read `~/.claude/plugins/` and list available skills from the system | `skills[]` entries |
| 2 | **Claude Code Agents** | Read `.claude/agents/` in project + `~/.claude/agents/` global | `agents.claude_code[]` entries |
| 3 | **SOVEREIGN Agents** | Call `mcp__council__list_agents` or `mcp__council__council_status` | `agents.sovereign[]` entries |
| 4 | **MCP Servers** | Read `~/.mcp.json` and list tools from each connected server | `mcp_tools[]` entries |
| 5 | **Project Dependencies** | Read `Cargo.toml`, `package.json`, `requirements.txt`, `go.mod` | `dependencies.installed[]` entries |
| 6 | **Existing Codebase Modules** | Glob for module files, read `lib.rs` / `index.ts` / `__init__.py` | Informal — used to inform spec challenge gate |

## Phase Mapping Rules

After discovery, each capability is mapped to SPEAR phases:

| Capability Pattern | Phase(s) | Timing |
|-------------------|----------|--------|
| Research, web search, analysis | spec, plan | on-demand |
| Code review, linting, testing | execute, audit | post-write, post-phase |
| Security scanning, vulnerability | audit | per-category |
| Code generation, scaffolding | execute | continuous |
| Documentation, writing | execute | post-write |
| Deployment, CI/CD | execute | post-phase |
| Performance profiling | audit | per-category |
| LLM routing, AI tasks | spec, plan, execute | on-demand |
| Memory, context management | all | continuous |
| Commit, git workflow | execute | at-commit |

## How Each Phase Uses the Registry

### Spec Phase
- **Before challenging requirements**: Check if existing capabilities already solve the requirement
- **Requirement Challenge Log**: "Can we solve this with what already exists?" references the registry
- **Deletion Proposal**: Cross-reference dead code against registered dependencies
- **Design approaches**: Propose solutions that leverage registered capabilities over building new

### Plan Phase
- **Task design**: Each task specifies which registered capability implements it
- **Fitness functions**: Registered testing/analysis tools are assigned to fitness functions
- **Risk assessment**: Missing capabilities are flagged as risks
- **Phase plan includes**: "Capabilities Used" section with registry references

### Execute Phase
- **Pre-execution**: Load registered capabilities for the current phase
- **During tasks**: Use registered skills/agents instead of manual implementation where available
- **Post-write**: Invoke registered code-review/testing capabilities
- **At-commit**: Use registered commit tools
- **Checkpoints**: Run registered fitness-check capabilities

### Audit Phase
- **Category enhancement**: Each audit category checks for registered tools that strengthen it:
  - Security: registered security scanners, SAST tools
  - Quality: registered linters, code reviewers
  - Performance: registered profilers
  - Architecture: registered type analyzers
- **Cross-validation**: Registered LLM providers used for ensemble auditing

### Ratchet Phase
- **Registry freshness check**: Flag if registry hasn't been refreshed in 3+ cycles
- **Capability utilization**: Track which registered capabilities were actually used vs. available
- **Retrospective**: Note unused capabilities that could have been leveraged

## Routing Decision Tree

When a task needs a capability:

```
1. Is there a registered SKILL that does this?
   YES → Use Skill tool (fastest, most integrated)
   NO → continue

2. Is there a registered SOVEREIGN AGENT for this domain?
   YES → Use mcp__council__invoke_agent (domain expertise)
   NO → continue

3. Is there a registered MCP TOOL that provides this?
   YES → Use the MCP tool directly
   NO → continue

4. Is there a registered Claude Code AGENT for this?
   YES → Use Agent tool with subagent_type
   NO → continue

5. Is there an installed DEPENDENCY that provides this?
   YES → Write code using the dependency
   NO → Build it (and consider if it should be a new dependency)
```

## Staleness & Refresh

- Registry is regenerated at the start of every SPEAR cycle (during `/spec`)
- Registry is refreshed during `/plan` (in case spec phase added new deps)
- Manual refresh: `/spear:toolkit refresh`
- If the registry is >3 cycles old: WARN in plan and execute phases
- If the registry doesn't exist: BLOCK — run `/spec` first

---

## Non-Blocking Principle

Registry lookup failures are **INFO-level, never blocking**. If a registered capability is unavailable at runtime (MCP server down, skill not installed, agent unreachable):

1. Log the failure
2. Fall back to manual implementation
3. Note in the deviation log
4. Flag in the audit for investigation
