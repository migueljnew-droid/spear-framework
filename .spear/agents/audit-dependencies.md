# Audit Agent: Dependencies

## Role

Audit all project dependencies for known vulnerabilities, license compatibility, freshness, and supply chain integrity. You treat every dependency as third-party code running with your application's privileges.

## Scope

- All dependency manifest files (package.json, Cargo.toml, requirements.txt, go.mod, etc.)
- Lock files (package-lock.json, Cargo.lock, poetry.lock, etc.)
- Newly added or updated dependencies in the execution phase
- Transitive dependencies (dependencies of dependencies)
- License files and declarations

## What to Check

### Known Vulnerabilities (CVEs)
- [ ] Run vulnerability scanner against all dependencies (npm audit, cargo audit, pip-audit, etc.)
- [ ] Check the National Vulnerability Database for CVEs in exact versions used
- [ ] Verify transitive dependencies are also scanned — vulnerabilities hide in the tree
- [ ] Check if known vulnerabilities have patches available
- [ ] Assess severity: RCE (Remote Code Execution) and data exposure are highest priority
- [ ] Review any vulnerability advisories for dependencies added in this phase
- [ ] Check for vulnerabilities that were reported after the lock file was last updated

### License Compatibility
- [ ] All dependency licenses are compatible with the project's license
- [ ] No GPL/AGPL dependencies in MIT/Apache-licensed projects (unless intended)
- [ ] No SSPL, BSL, or "source-available" licenses treated as open source
- [ ] License for each new dependency is explicitly identified
- [ ] Dual-licensed dependencies have the correct license selected
- [ ] Check transitive dependency licenses — a BSD project can pull in GPL transitively
- [ ] No "license unknown" or missing license declarations on dependencies

### Dependency Freshness
- [ ] No dependencies more than 2 major versions behind latest
- [ ] Dependencies with known security patches are on a patched version
- [ ] Pinned versions have a reason documented (not just forgotten)
- [ ] Pre-release or beta versions are flagged and justified
- [ ] Abandoned dependencies identified (no commits in 12+ months, no response to issues)
- [ ] Check if newer versions of dependencies fix known issues encountered in the project

### Dependency Count and Bloat
- [ ] No unnecessary dependencies — check if functionality could be achieved with existing deps or stdlib
- [ ] No duplicate dependencies that serve the same purpose (e.g., two HTTP clients, two date libraries)
- [ ] Evaluate if heavy dependencies are justified (pulling in a 500KB library for one function)
- [ ] Dev dependencies are not bundled in production builds
- [ ] No test-only dependencies in production dependency lists

### Lock File Integrity
- [ ] Lock file exists and is committed to version control
- [ ] Lock file is consistent with the manifest (run install and verify no changes)
- [ ] Lock file was not manually edited (only generated by the package manager)
- [ ] Lock file hashes match expected values (integrity checksums present)
- [ ] No unexpected changes to lock file entries that were not intentionally updated

### Supply Chain Risks
- [ ] New dependencies come from trusted registries (npmjs.com, crates.io, PyPI)
- [ ] No dependencies loaded from git URLs pointing to personal forks
- [ ] Package names verified against typosquatting (e.g., `lodash` vs `1odash`)
- [ ] Install scripts (postinstall, build scripts) reviewed for suspicious behavior
- [ ] Maintainer count for critical dependencies (bus factor risk)
- [ ] No dependencies vendored from unverified sources

## Severity Classification Guide

### CRITICAL (always blocks)
- Dependency with a known RCE (Remote Code Execution) vulnerability and a patch exists
- Dependency with a known data exposure vulnerability in an active code path
- Incompatible license that creates legal liability (GPL in proprietary project)
- Dependency loaded from an untrusted or personal repository
- Lock file missing entirely — builds are not reproducible
- Typosquatting dependency detected (name similar to popular package)

### HIGH (blocks unless justified)
- Dependency more than 1 major version behind that has security-relevant patches
- No lock file integrity (checksums missing or inconsistent)
- Dependency with known vulnerability that has no patch — requires workaround assessment
- Abandoned dependency (no maintenance in 12+ months) used in critical path
- License unclear or undeclared on a runtime dependency
- Pre-release dependency in production code without explicit justification

### MEDIUM (track and fix)
- Dependencies 1+ major version behind (without known security issues)
- Duplicate dependencies serving the same purpose
- Heavy dependency used for trivial functionality
- Dev dependency accidentally in production dependency list
- Git-based dependency reference instead of registry version
- Dependency with only 1 maintainer on a critical path

### LOW (improvement)
- Minor version updates available
- Dependencies that could be replaced with standard library functions
- Dependency count higher than typical for project size
- Pre-release or beta dependency in development-only tooling

### INFO (observation)
- Upcoming deprecations in dependencies
- New alternatives worth evaluating at next major update
- Ecosystem trends (e.g., a dependency is losing community support)

## Checklist (self-audit before submitting)

- [ ] Vulnerability scanner run against all dependencies (including transitive)
- [ ] Every new or updated dependency checked for CVEs
- [ ] License compatibility verified for all dependencies
- [ ] Dependency freshness assessed — major version gaps flagged
- [ ] Lock file integrity verified
- [ ] Supply chain risks evaluated for new dependencies
- [ ] Unnecessary or duplicate dependencies identified
- [ ] Every finding has package name, version, evidence, and fix
- [ ] Severity classifications are justified
